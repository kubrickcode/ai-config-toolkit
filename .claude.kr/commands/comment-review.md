# 주석 리뷰 커맨드

## 목적

불필요한 주석을 공격적으로 식별하고 제거한다. 코드가 스스로 말해야 한다.

**핵심 원칙**: 코드에 주석이 필요하면, 그 코드가 잘못된 것이다. 코드를 먼저 고쳐라.

**기본 자세**: 의심스러우면 제거(REMOVE). 주석은 기술 부채다.

---

## 사용자 입력

```text
$ARGUMENTS
```

**해석 우선순위**:

1. $ARGUMENTS 있음 → 사용자 요청 따르기 (파일 경로, `--all` 전체 프로젝트 등)
2. 비어있음 → `git status`로 커밋되지 않은 변경사항 확인
3. 변경사항 없음 → 최신 커밋 검토
4. 전체 프로젝트는 명시적 요청 시에만 (`--all` 또는 "전체 코드베이스 검토")

---

## 주석 원칙

### 🚫 주석의 문제점

- **주석은 썩는다**: 코드는 바뀌는데 주석은 안 바뀜. 결과: 코드베이스에 거짓말.
- **주석은 나쁜 코드를 가린다**: 주석이 필요하면 이름이나 구조가 잘못된 것.
- **주석은 유지보수 부담**: 모든 주석은 동기화해야 할 코드.

**경험법칙**: 주석을 쓰려면 멈추고 "어떻게 더 나은 코드로 이걸 없앨 수 있을까?"를 물어라.

### ✅ 유일하게 합법적인 주석

이것들이 주석이 허용되는 **유일한** 경우. 나머지는 모두 제거하거나 리팩토링해야 한다.

| 카테고리              | 엄격한 기준                                 | 유효한 예시                                                   |
| --------------------- | ------------------------------------------- | ------------------------------------------------------------- |
| **공개 API 문서화**   | 외부 공개 API만. 비공개 코드 = 문서 불필요. | `@param userId - 고유 식별자` (내부 헬퍼 함수에는 ❌)         |
| **비즈니스 로직 WHY** | 도메인 지식이 코드 외부에 있을 때만         | "GDPR 17조에 따라 데이터는 7년간 보존해야 함"                 |
| **외부 의존성**       | 제어 불가능한 서드파티 API 제약             | "Stripe webhook은 지수 백오프로 3회 재시도" (문서 URL 명시)   |
| **성능/보안 경고**    | 리뷰어를 놀라게 할 반직관적 결정            | "의도적 O(n²) - 도메인 요구사항에 따라 n은 5를 초과하지 않음" |
| **복잡한 패턴**       | 패턴 자체가 본질적으로 난해할 때만 (드뭄)   | 정규식 분해 - 그러나 먼저 정규식 단순화 시도                  |
| **법적/라이선스**     | 저작권 헤더, 라이선스 귀속                  | "MIT License", "특허 US1234567" (법적 필수)                   |
| **TODO/FIXME**        | 필수: 담당자, 티켓, 마감일 또는 제거 조건   | `// TODO(@username): #1234 Q2 마이그레이션 완료 후 제거`      |

### ⚠️ 비즈니스 로직 주석 규칙

**단일 진실 공급원**: 비즈니스 로직 설명은 **정확히 한 곳에만** 존재해야 한다.

- ❌ **금지**: 같은 설명이 여러 파일에 분산
- ✅ **필수**: 도메인 모델/서비스에 한 번 문서화, 명확한 함수명으로 참조

**예시**:

```typescript
// ❌ 나쁨: 여기저기 반복
function validateAge(age) {
  // 법적 요구사항에 따라 사용자는 18세 이상이어야 함
  return age >= 18;
}

function checkEligibility(user) {
  // 법적 요구사항에 따라 사용자는 18세 이상이어야 함
  if (user.age < 18) return false;
}

// ✅ 좋음: 한 번 문서화, 다른 곳은 자기 문서화 코드
/**
 * 법적 최소 연령 (관할권 요구사항)
 * @see docs/compliance/age-verification.md
 */
const LEGAL_MINIMUM_AGE = 18;

function isLegalAge(age: number): boolean {
  return age >= LEGAL_MINIMUM_AGE;
}
```

## 분류 기준

### ❌ REMOVE (즉시 삭제)

**기본 가정**: 모든 주석은 위의 합법적 사례에 정확히 해당하지 않으면 제거해야 한다.

| 유형                 | 예시                                              | 왜 잘못됐나                       | 해결책                                       |
| -------------------- | ------------------------------------------------- | --------------------------------- | -------------------------------------------- |
| **코드 흐름 서술**   | `// 사용자들을 순회`                              | 코드가 이미 말하고 있음           | 삭제 - `for (const user of users)`가 명확함  |
| **자명한 설명**      | `// 카운터 증가` (위에 `counter++`)               | 독자 지능 모독                    | 삭제                                         |
| **이름 보상**        | `// 사용자 데이터 가져오기` (위에 `getData()`)    | 함수명이 잘못됨                   | `getUserData()`로 이름 변경, 주석 삭제       |
| **WHAT 대신 WHY**    | `// 이메일 형식 검증`                             | 함수명이 이걸 전달해야 함         | `isValidEmailFormat()`으로 이름 변경, 삭제   |
| **타입 중복**        | `// 문자열 반환` (TypeScript)                     | 타입 시스템이 이미 선언함         | 삭제 - 반환 타입과 중복                      |
| **닫는 괄호 라벨**   | `} // end if`                                     | 들여쓰기가 구조를 보여줌          | 삭제 - 올바른 포맷팅이면 이게 명확함         |
| **섹션 구분자**      | `// ===== Validation =====`                       | 함수가 너무 크다는 신호           | 별도 함수로 추출, 구분자 삭제                |
| **주석 처리된 코드** | `// oldFunction()`                                | Git 히스토리가 있는 이유가 있음   | 삭제 - 예전 코드 필요하면 `git log` 사용     |
| **오래된/거짓**      | 주석이 실제 코드와 모순                           | 위험한 오정보                     | 즉시 삭제 - 이것이 버그를 유발함             |
| **변수 재진술**      | `const userId = 1; // 사용자 ID`                  | 변수명이 이미 말하고 있음         | 삭제                                         |
| **파라미터 설명**    | `calculate(x, y) // x는 너비, y는 높이`           | 파라미터 이름이 잘못됨            | 파라미터를 `width, height`로 이름 변경, 삭제 |
| **반환값 설명**      | `return result; // 결과 반환`                     | 무의미한 동어반복                 | 삭제                                         |
| **import 정당화**    | `import lodash from 'lodash'; // 유틸리티 함수용` | import문이 자명함                 | 삭제                                         |
| **잡음 단어**        | `// 중요!`, `// 참고:`, `// 기억:`                | 중요하면 코드가 반영해야 함       | 코드 구조 수정, 주석 삭제                    |
| **역사적 코멘터리**  | `// 2023-01-15 홍길동 추가`                       | Git blame이 이걸 보여줌           | 삭제 - 히스토리는 `git log` 사용             |
| **할당 설명**        | `isValid = true; // valid를 true로 설정`          | 할당이 자명함                     | 삭제                                         |
| **희망적 사고**      | `// 이게 작동할 거야`                             | 불확실함은 프로덕션에 있으면 안됨 | 제대로 테스트하고 삭제                       |
| **디버깅 잔재**      | `// console.log(x);`                              | 디버깅 세션의 잔재                | 삭제 - 적절한 디버깅 도구 사용               |
| **언어 기능 서술**   | `// async/await 사용`                             | 코드가 말 그대로 이걸 보여줌      | 삭제                                         |

**공격적 규칙**: 주석이 코드가 **무엇을(WHAT)** 하는지 설명하면 (**왜(WHY)**가 아니라), 반드시 제거해야 한다. 코드 자체가 WHAT을 설명해야 한다.

### ⚠️ IMPROVE (리팩토링 필요)

이 카테고리의 주석은 **일시적으로 허용**되지만 개선해야 한다. 목표: 리팩토링으로 제거.

| 유형                     | 문제                      | 필수 조치                                         |
| ------------------------ | ------------------------- | ------------------------------------------------- |
| **불명확한 TODO**        | `// TODO: 나중에 고치기`  | **필수**: 담당자, 티켓, 조건 추가                 |
| **WHAT 설명**            | `// 사용자 입력 검증`     | `validateUserInput()`으로 이름 변경, 삭제         |
| **장황한 설명**          | 여러 단락의 주석 블록     | 문서 파일로 추출, 코드 주석에서 링크              |
| **비표준 문서**          | 공개 API에 일반 주석      | JSDoc/TSDoc/GoDoc으로 변환, 아니면 API가 아님     |
| **분산된 비즈니스 로직** | 같은 규칙이 파일들에 중복 | 한 곳에 통합, 이름으로 참조                       |
| **컨텍스트 없는 TODO**   | `// FIXME: 깨짐`          | 티켓 번호, 재현 단계 추가                         |
| **모호한 WHY**           | `// 성능 때문에`          | 구체화: "N+1 쿼리 방지용 캐시 (#1234 참조)"       |
| **인용 누락**            | `// API 제한 적용됨`      | URL 추가: "API 속도 제한: 100/분 (docs.com 참조)" |

**리팩토링 우선순위**:

1. **먼저**, 코드 개선으로 제거 시도
2. **불가능할 때만**, 주석 품질 개선
3. **마감 설정**: 모든 IMPROVE 주석은 REMOVE 또는 KEEP이 되는 계획 필요

### ✅ KEEP (합법적으로 필요)

**엄격한 기준** - 이것들이 모두 참이어야 함:

- ✅ 위의 합법적 카테고리 중 하나에 해당
- ✅ WHAT이 아닌 WHY 설명
- ✅ 코드만으로 표현 불가
- ✅ 간결함 (이상적으로 한 줄, 최대 3줄)
- ✅ 코드와 동기화됨 (정확성 검증됨)
- ✅ 단일 진실 공급원 (다른 곳에 중복 없음)

**검증 질문** (하나라도 "아니오"면 주석 제거):

1. 코드 외부의 것을 설명하는가 (비즈니스 규칙, API 제약, 법적 요구사항)?
2. 능숙한 개발자가 이 주석 없이 진정으로 놀랄 것인가?
3. 코드 구조/이름으로 표현하는 모든 옵션을 다 썼는가?
4. 이 정보가 코드베이스에서 이곳에만 존재하는가?

## 워크플로우

### 1. 분석 대상 결정

```bash
git status --porcelain
```

**결정 트리**:

1. $ARGUMENTS에 특정 요청 있음 → 요청 따르기
2. $ARGUMENTS 비어있음 + 커밋되지 않은 변경 있음 → `git diff`로 변경 파일 분석
3. $ARGUMENTS 비어있음 + 변경 없음 → `git show HEAD`로 최신 커밋 분석
4. 사용자가 전체 프로젝트 명시적 요청 (`--all`) → 전체 프로젝트 스캔

### 2. 주석 추출

**언어별 패턴**:

- TypeScript/JavaScript: `//`, `/* */`, `/** */`
- Go: `//`, `/* */`
- Python: `#`, `""" """`
- HTML/JSX: `{/* */}`, `<!-- -->`

**각 주석에 포함**:

- 주석 내용
- 파일 경로:라인번호
- 주변 코드 컨텍스트 (±3줄)

### 3. 공격적 편향으로 분류

**결정 알고리즘**:

```
각 주석에 대해:
  1. 7개 합법적 카테고리 중 하나에 정확히 해당하는가?
     아니오 → REMOVE

  2. 정보를 더 나은 코드로 표현할 수 있는가?
     예 → REMOVE (리팩토링 제안)

  3. WHY 대신 WHAT을 설명하는가?
     예 → REMOVE

  4. 코드베이스의 다른 곳에 중복되어 있는가?
     예 → REMOVE (한 곳에 통합)

  5. 모호하거나, 오래됐거나, 컨텍스트가 없는가?
     예 → IMPROVE (고칠 가치 없으면 REMOVE)

  6. 모든 검사 통과?
     → KEEP (그래도 정말 필요한지 검증)
```

**제거 편향**: 모호한 경우, 리팩토링 제안과 함께 REMOVE 선호.

### 4. 조치 결정

| 상황                                                 | 조치                                                |
| ---------------------------------------------------- | --------------------------------------------------- |
| **명백한 제거 대상** (주석 처리된 코드, 자명한 설명) | **즉시 제거** - 논의 불필요                         |
| **리팩토링 필요** (이름 개선, 추출)                  | **리팩토링 제안** - before/after 보여주기           |
| **의도 불명확** (합법성 판단 불가)                   | **리뷰 플래그** - 사용자에게 비즈니스 컨텍스트 문의 |
| **전체 프로젝트 리뷰**                               | **리포트 생성** - 공격적 리팩토링 권장 포함         |

**톤**: 직접적이고 단호하게. 메시지를 부드럽게 하지 마라.

- ❌ 약함: "이 주석이 필요하지 않을 수도 있습니다"
- ✅ 강함: "제거 - 이 주석은 중복입니다. 코드가 자명합니다."

## 출력 형식

### 개별 파일 / 변경사항 (직접 조치)

즉각적이고 실행 가능한 피드백 제공:

```markdown
## 주석 리뷰: {file_path}

### ❌ REMOVE - 즉시 삭제 ({count})

**라인 {n}**: `{comment}`
**이유**: {간단한 근거}
**조치**: 삭제됨

**라인 {n}**: `{comment}`
**이유**: 잘못된 이름 보상
**리팩토링**:

- `{old}`를 `{new}`로 이름 변경
- 주석 삭제

### ⚠️ IMPROVE 또는 제거 ({count})

**라인 {n}**: `{comment}`
**문제**: {이슈}
**옵션**:

1. `{better_code}`로 리팩토링하고 주석 삭제
2. 리팩토링 불가시, 다음으로 개선: `{better_comment}`
   **권장**: 옵션 1

### ✅ KEEP ({count})

**라인 {n}**: `{comment}` - 외부 API 제약, 적절히 문서화됨
```

### 전체 프로젝트 리뷰 (공식 리포트)

전체 프로젝트 리뷰 시에만 생성 (`--all`):

````markdown
# 📝 주석 리뷰 리포트

**생성일**: {timestamp}
**범위**: 전체 프로젝트
**분석된 주석 수**: {count}
**자세**: 공격적 제거 - 코드가 자기 문서화되어야 함

---

## 📊 요약

| 카테고리   | 수량 | 비율 | 상태                     |
| ---------- | ---- | ---- | ------------------------ |
| ❌ REMOVE  | {n}  | {%}  | 즉시 삭제                |
| ⚠️ IMPROVE | {n}  | {%}  | 제거를 위해 리팩토링     |
| ✅ KEEP    | {n}  | {%}  | 합법적으로 필요함 확인됨 |

**정리 잠재력**: {remove + improve}개 주석 ({%})을 리팩토링으로 제거 가능.

````

## 실행 지침

### 마인드셋: 공격적 제거

**당신은**: 코드 품질 집행자. 기술 부채를 제거하는 게 당신의 일.
**당신은 아님**: 외교관. 메시지를 부드럽게 하지 마라. 직접적으로.

**기본 가정**: 모든 주석은 결백이 증명될 때까지 유죄.

### 단계별 프로세스

1. **입력 파싱**: $ARGUMENTS 해석

2. **범위 결정**:
   - 사용자 요청 → 정확히 따르기
   - 비어있음 → `git status --porcelain` 확인
   - 변경 있음 → 변경 파일에 `git diff` 사용
   - 변경 없음 → 최신 커밋에 `git show HEAD` 사용

3. **주석 추출**: 대상 파일에서 모든 주석 패턴 검색

4. **편향으로 분류**:
   - **1차 패스**: 모든 주석을 REMOVE로 표시
   - **2차 패스**: 어떤 주석이 IMPROVE나 KEEP이어야 하는지 정당화
   - **3차 패스**: 모든 KEEP에 도전 - 리팩토링으로 제거할 수 있는가?

5. **단호한 조치**:
   - **REMOVE 카테고리**: 즉시 삭제, 허락 불필요
   - **IMPROVE 카테고리**: 리팩토링 코드 제공, 개선보다 삭제 강력 권장
   - **KEEP 카테고리**: 정말 모든 기준을 충족하는지 검증
   - **UNCLEAR 카테고리**: 조치 전 반드시 사용자에게 질문

6. **사용자에게 질문해야 할 때** (필수):
   - 주석이 이해할 수 없는 도메인 지식을 참조할 때
   - 주석이 다른 곳에 문서화되지 않은 비즈니스 규칙을 언급할 때
   - 컨텍스트 분석 후에도 주석 목적이 진정으로 애매할 때
   - 주석 제거가 암묵적 계약을 깨뜨릴 수 있을 때
   - **형식**: "이 주석은 [X]를 언급합니다. 아직 유효한가요? 제거할까요, 개선할까요?"

7. **리팩토링 제공**:
   - 구체적인 before/after 코드 보여주기
   - 구현 시간 추정
   - 고영향 리팩토링 우선순위화

8. **강한 언어 사용**:
   - ❌ "이 주석 제거를 고려해보세요"
   - ✅ "제거 - 이 주석은 기술 부채입니다"
   - ❌ "이게 불필요할 수도 있습니다"
   - ✅ "삭제 - 코드가 자명합니다"

### 언어별 고려사항

- **TypeScript/JavaScript**: JSDoc은 export된 공개 API에만. 비공개 함수는 문서 불필요.
- **Go**: GoDoc은 export된 심볼에만. unexported = 문서 불필요.
- **Python**: Docstring은 공개 모듈/클래스/함수에만.

**규칙**: 공개 API가 아니면 문서화 주석이 필요 없다. 코드가 자명해야 한다.
